# Lifting Log Foundation Milestone Design

**Date:** 2026-05-04

**Goal:** Replace the current mock-data SwiftUI prototype with a durable iOS 17+ SwiftData foundation while preserving the existing dark workout-tracker UI skeleton where reasonable.

## Current Project Audit

The project is a native SwiftUI app generated from `project.yml` into `LiftingLog.xcodeproj`.

- App target: `LiftingLog`
- Unit test target: `LiftingLogTests`
- UI test target: `LiftingLogUITests`
- Minimum deployment target: iOS 17.0
- Swift version: 6.0
- Verified local toolchain: Xcode 26.4.1, XcodeGen 2.45.4

The current implementation is prototype-grade and uses in-memory state:

- `LiftingLog/App/LiftingLogApp.swift` creates `AppStore.preview`.
- `LiftingLog/App/AppShellView.swift` switches between History, Workout, and Profile tabs.
- `LiftingLog/Shared/Stores/AppStore.swift` owns active workout state, history state, and mutation methods.
- `LiftingLog/Shared/Models/WorkoutModels.swift` defines value-type mock models.
- `LiftingLog/Shared/Mocks/MockRepository.swift` provides mock active workout and mock history.
- No SwiftData, HealthKit, Watch, Live Activities, Dynamic Island, App Intents, widgets, or CloudKit code exists.

The repo currently has user-owned dirty UI changes. The implementation should read and preserve them, not revert them.

## Product Scope

Milestone 1 ships a useful offline workout tracker without relying on native integrations or cloud sync.

Included:

- SwiftData-backed exercise library with seed exercises.
- Start workout flow.
- Option to start a blank workout.
- Option to use a past completed workout as the starting point for a new workout.
- Active workout screen wired to persisted data.
- Add/remove logged exercises.
- Add/remove logged sets.
- Edit weight, reps, RPE, exercise notes, and workout notes.
- Finish workout into history.
- Delete completed workouts from history.
- Exercise history list derived from logged data.
- Basic exercise create/edit/archive flow for custom exercises.
- Basic settings model and profile/settings screen.
- Tests for model creation, persistence, seed data, settings, and workout logging behavior.

Deferred:

- HealthKit sync and permissions.
- Apple Watch app.
- Live Activities and Dynamic Island.
- App Intents and Shortcuts.
- Widgets.
- CloudKit sync.
- Charts, PR analytics, and advanced progression planning.
- Dedicated named workout template management UI.

## Clarified Template Decision

There should not be a first-class template product surface in this milestone.

Instead, completed workout sessions are reusable starting points. The start-workout screen should offer:

- Blank Workout
- Use Past Workout

Starting from a past workout creates a new active workout with the same exercise order and set structure. The new logged sets should be incomplete. Previous weight, reps, and RPE can be copied into the editable fields as convenient starting values because lifters commonly repeat or adjust the prior prescription.

The data layer may include a future-facing `WorkoutTemplate` model or a `WorkoutBlueprint` value abstraction, but no named-template CRUD screen should ship in milestone 1.

## SwiftData Model Boundaries

All persistent models should have stable UUIDs separate from SwiftData identity.

### Exercise

Represents the exercise library.

Fields:

- `id: UUID`
- `seedIdentifier: String?`
- `name: String`
- `categoryRaw: String`
- `equipmentRaw: String`
- `primaryMuscleRaw: String`
- `notes: String`
- `isArchived: Bool`
- `isSeeded: Bool`
- `createdAt: Date`
- `updatedAt: Date`

Rules:

- Seeded exercises are inserted idempotently.
- Custom exercises can be created and edited.
- Delete should archive by default if the exercise has logged history.
- Hard delete is acceptable only for custom exercises with no logged sets.

### WorkoutTemplate

Future-facing model for named templates. It is included to satisfy the original model boundary, but it should not drive the milestone 1 UI.

Fields:

- `id: UUID`
- `name: String`
- `notes: String`
- `createdAt: Date`
- `updatedAt: Date`
- `isArchived: Bool`
- `healthLinkID: UUID?`

Milestone 1 should not add template list, template builder, or template CRUD screens. The active product flow uses past workouts as templates through `WorkoutBlueprint`.

### WorkoutSession

Represents both active and completed workout sessions.

Fields:

- `id: UUID`
- `title: String`
- `startedAt: Date`
- `endedAt: Date?`
- `durationSeconds: Int`
- `notes: String`
- `statusRaw: String`
- `sourceRaw: String`
- `sourceSessionID: UUID?`
- `createdAt: Date`
- `updatedAt: Date`
- `healthLinkID: UUID?`
- relationship `loggedExercises: [LoggedExercise]`

Rules:

- Only one session should have `statusRaw == "active"`.
- Completed sessions appear in history.
- Discarded sessions are hidden from normal history.

### LoggedExercise

Represents one exercise inside one workout session.

Fields:

- `id: UUID`
- `orderIndex: Int`
- `exerciseSnapshotName: String`
- `notes: String`
- `createdAt: Date`
- `updatedAt: Date`
- relationship `exercise: Exercise?`
- relationship `session: WorkoutSession?`
- relationship `sets: [LoggedSet]`

Rules:

- Keep `exerciseSnapshotName` so history remains readable if the library exercise is renamed or archived.
- Order is explicit and stable.

### LoggedSet

Represents one set inside one logged exercise.

Fields:

- `id: UUID`
- `orderIndex: Int`
- `weight: Double?`
- `reps: Int?`
- `rpe: Double?`
- `kindRaw: String`
- `isCompleted: Bool`
- `completedAt: Date?`
- `notes: String`
- `createdAt: Date`
- `updatedAt: Date`
- `healthLinkID: UUID?`
- relationship `loggedExercise: LoggedExercise?`

Rules:

- Completion is explicit and independent from whether weight/reps fields are non-empty.
- Volume is computed from completed sets with both weight and reps.

### UserSettings

Singleton app settings record.

Fields:

- `id: UUID`
- `weightUnitRaw: String`
- `defaultRestTimerSeconds: Int`
- `hasCompletedOnboarding: Bool`
- `createdAt: Date`
- `updatedAt: Date`

Rules:

- `SeedDataService` or `SettingsStore` creates exactly one settings record if missing.
- First milestone needs only persisted unit settings and a default rest timer setting.

### HealthDataLink

Future extension point. It must not import HealthKit yet.

Fields:

- `id: UUID`
- `providerRaw: String`
- `localEntityKindRaw: String`
- `localEntityID: UUID`
- `externalIdentifier: String`
- `externalType: String`
- `syncStatusRaw: String`
- `lastSyncedAt: Date?`
- `createdAt: Date`
- `updatedAt: Date`

Rules:

- Future HealthKit work should use `providerRaw == "healthKit"`.
- Keep it generic so HealthKit samples, workouts, and future providers do not force core model redesign.

### SeedMetadata

Tracks seed versions.

Fields:

- `id: UUID`
- `key: String`
- `version: Int`
- `appliedAt: Date`

Rules:

- Use key `exerciseSeed`.
- Seed service can be rerun safely.

## Domain Types

Use String-backed enums for persisted raw values:

- `ExerciseCategory`: strength, cardio, mobility, other
- `ExerciseEquipment`: barbell, dumbbell, machine, cable, bodyweight, kettlebell, other
- `MeasurementUnit`: pounds, kilograms
- `WorkoutSessionStatus`: active, completed, discarded
- `WorkoutSource`: blank, pastWorkout, template
- `SetKind`: working, warmup, drop, failure
- `HealthSyncStatus`: notSynced, synced, failed

Use value types for computed or transient behavior:

- `WorkoutBlueprint`: title, notes, exercises, sets; created from blank state, past sessions, and later named templates.
- `WorkoutMetrics`: completed set count, total set count, completed volume, duration.

## Active Workout Engine

Create `ActiveWorkoutEngine` as a root-owned `@Observable` object. It should coordinate active workout mutations against a `ModelContext`.

Responsibilities:

- Load the current active session on launch.
- Start a blank workout.
- Start from a past completed workout.
- Add and remove logged exercises.
- Add and remove logged sets.
- Update set weight, reps, RPE, completion, and notes.
- Update workout title and notes.
- Finish a workout.
- Discard a workout.
- Compute active workout metrics.
- Enforce a single active workout.

The engine should store only lightweight UI state:

- `activeSessionID: UUID?`
- `isStartingWorkout: Bool`
- `lastErrorMessage: String?`

It should not duplicate the full SwiftData object graph in memory. Views should query the active `WorkoutSession` and pass model objects or IDs into focused engine methods.

The timer should be derived from `startedAt` while active, using a SwiftUI timer or `TimelineView`. Do not persist a tick every second. Persist `durationSeconds` when finishing.

## Navigation

Root navigation uses the existing three-tab shell:

- History
- Workout tab
- Profile

The workout tab is state-aware:

- No active workout: title `Start`, icon `plus.circle`, destination `StartWorkoutView`
- Active workout: title `Current`, icon `timer`, destination `WorkoutSessionView`

`StartWorkoutView` includes:

- Primary action: blank workout
- Secondary section: recent completed workouts to reuse
- Empty state if no history exists

Selecting a past workout creates a new active session from that completed session and routes immediately to the current workout screen.

## Testing Strategy

Use unit tests with in-memory SwiftData containers for most behavior. Keep UI tests as smoke tests.

Test groups:

- Model creation and relationships.
- Persistence save/fetch with in-memory container.
- Seed service idempotency and expected exercises.
- Settings singleton creation.
- Active workout start blank.
- Active workout start from past workout.
- Active workout add exercise and set.
- Set completion and metrics.
- Finish workout moves session to history.
- Delete workout removes or hides history item.
- UI smoke: launch, start blank workout, finish or keep going, navigate history/profile.

## Acceptance Criteria

- App launches with SwiftData-backed persistence.
- Seed exercises exist on first launch and are not duplicated on subsequent launches.
- User can start a blank workout.
- User can start a workout from a past completed workout.
- Workout tab label/icon changes when an active workout exists.
- Active workout edits persist across tab switches and app relaunch.
- User can finish a workout and see it in history.
- History and exercise history are driven by persisted data.
- User can create/edit/archive custom exercises.
- Settings are persisted.
- Unit and UI tests pass with the commands above.
- No HealthKit, Watch, Live Activities, Dynamic Island, App Intents, widgets, or CloudKit code is required.
