Clerk Authentication Design

Phase 3 implementation plan for adding optional Clerk identity to Lifting Log without interrupting the local, offline workout logging experience.

Issue 6 Date: 2026-05-31 Environment: Clerk Development UI: Clerk prebuilt views

Decision

Use Clerk's prebuilt iOS authentication UI for v1. Authentication is optional: signed-out users can continue using the app locally, and signed-in users get an account identity that future Convex sync can use.

The app will support email/password and Sign in with Apple based on the Clerk Dashboard configuration. Google, phone, username, account deletion, and sync are outside this issue.

Goals

  • Add sign in, sign out, and session restoration through Clerk.
  • Keep all existing workout logging, history, exercise library, settings, and export flows available while signed out.
  • Make signed-out behavior intentional and visible instead of treating it as an error.
  • Place account state in Profile, with an optional matching account entry in Settings if it improves discoverability.
  • Keep v1 auth implementation small enough to verify manually on a physical device.

Non-Goals

  • No Convex sync or cloud data migration in this issue.
  • No account deletion flow; that remains Phase 6 before App Store submission.
  • No custom email, password reset, or Apple sign-in screens.
  • No Google provider setup for v1.
  • No forced login gate on app launch.

Architecture

The existing SwiftData app shell remains the root experience. LiftingLogApp configures Clerk.shared once at launch and injects it into the SwiftUI environment. Profile and Settings read Clerk state from the environment, while workout and persistence features remain independent of Clerk.

App launch: configure Clerk with the current publishable key and restore any existing session.
Signed out: show local profile stats and a sign-in account card. All local data features remain available.
Sign in: present AuthView() as a dismissible sheet from Profile or Settings.
Signed in: show basic account state and UserButton, which opens Clerk's profile/account sheet and handles sign-out.
Sign out: return Profile to local signed-out mode. Local SwiftData records remain on device.

Profile Experience

Profile remains the main account surface. The hardcoded local name should be replaced by account-aware content:

  • When signed out, show a neutral local-mode label such as "Local lifting log" and a primary action to sign in for backup and sync.
  • When signed in, show the user's available Clerk identity, such as primary email address, and expose UserButton for account management.
  • The stats cards and app data summary should not depend on auth state.
  • Exercise Library and Settings remain reachable in both states.

Settings Experience

Settings should continue to focus on app preferences and export. If account discoverability needs a second entry point, add a compact Account section that mirrors Profile: signed out users can open AuthView(), signed in users can open Clerk account controls.

The implementation can start with Profile only. Settings should not become the primary auth surface unless Profile proves insufficient during manual testing.

Development And Production Environments

Issue 6 will use the current Clerk Development instance and pk_test_... publishable key. That is appropriate for local development and physical-device testing.

Before TestFlight release-candidate hardening and before App Store review, create and configure the Clerk Production instance. The production pass must use pk_live_..., the production Frontend API domain, the production iOS native application registration, and production Associated Domains.

Production should be created no later than Phase 4 when Clerk identity is connected to Convex auth, because backend auth configuration should be validated against the same environment intended for release.

Configuration Notes

  • The Associated Domains entitlement should use webcredentials:glad-krill-22.clerk.accounts.dev for the development instance, without https://.
  • The publishable key should be isolated behind a small configuration boundary before production switching, even if it remains hardcoded for the first local implementation pass.
  • Clerk package products ClerkKit and ClerkKitUI should stay linked only to the app target unless tests directly import Clerk.

Error Handling

  • Auth sheet dismissal is allowed and returns the user to local mode.
  • Clerk UI handles email verification, password reset, Apple sign-in, and recoverable auth errors.
  • If Clerk cannot restore a session, the app should still launch into signed-out local mode.
  • Sign-out must not delete local workout history or settings.

Testing

Automated

  • Keep existing workout, history, settings, export, and UI tests passing while Clerk is configured.
  • Add focused tests only for local logic that does not require live Clerk services, such as display-state helpers if introduced.

Manual Device Verification

  1. Launch signed out and start, edit, finish, and view a workout.
  2. Open Profile and present the Clerk auth sheet.
  3. Dismiss the sheet and confirm local mode still works.
  4. Sign in with email/password and verify Profile reflects signed-in state.
  5. Sign out with Clerk account controls and verify local data remains.
  6. Relaunch the app and verify session restoration for a signed-in user.
  7. After Apple Developer Program enrollment is complete, verify Sign in with Apple on a physical device.

Implementation Sequence

  1. Fix configuration details and introduce a small Clerk configuration boundary.
  2. Add an auth/account component that renders signed-out and signed-in states from Clerk.shared.
  3. Integrate that component into Profile.
  4. Optionally add a compact Settings account section if Profile-only feels hidden.
  5. Build and run unit/UI tests, then manually verify on device.
