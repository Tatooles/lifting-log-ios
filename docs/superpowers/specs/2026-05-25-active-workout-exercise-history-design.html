Active Workout Exercise History Shortcut Design

Date: 2026-05-25 · Issue: GitHub #21, “Add active-workout shortcut to exercise history”

Summary

Add a clear history affordance to each active workout exercise card so users can review recent completed history for that exercise without losing their workout context. The first tap opens a read-only quick-history sheet. The sheet emphasizes recent session data and provides a full-history action that deep-links into the History tab’s canonical exercise detail screen.

Goals

  • Let users quickly reference past performance for the exercise they are currently logging.
  • Keep the active workout available underneath the quick-history sheet.
  • Expose a path to the full exercise history page in the History tab.
  • Handle empty history gracefully.
  • Stay narrowly scoped and avoid reworking the full History tab architecture beyond the routing needed for deep linking.

Non-Goals

  • No editing, deleting, or mutation of historical workout data from the quick sheet.
  • No charts, personal-record calculations, trend analysis, or best-set metrics in this issue.
  • No redesign of the History tab’s list or detail pages.
  • No workout-level notes in the quick sheet. Exercise-level notes are included because the shortcut is exercise-specific.

Chosen Approach

Use a dedicated quick-history sheet launched from the active exercise card header, backed by existing exercise history grouping logic where practical. “View Full History” should switch to the History tab, select Exercises mode, and open the matching exercise detail.

This approach keeps the logging loop fast while preserving the History tab as the canonical full-history surface. It also supports the long-term UX where users can move between their active workout and the selected exercise’s full history using the tab bar.

Architecture

Workout Entry Point

WorkoutSessionView owns presentation state for the selected LoggedExercise. ExerciseCardView receives a history action closure and renders a dedicated button in the card header near the set-count badge and delete action. The exact SF Symbol can be chosen during implementation, but the button must have an explicit accessibility label such as “View exercise history” and a stable UI-test identifier.

Quick-History Sheet

Add a new sheet view, tentatively ExerciseQuickHistorySheet. It receives the current LoggedExercise and a deep-link callback. It queries completed workout sessions, derives the matching exercise history, and displays up to three most recent completed session groups.

Full-History Deep Link

Extend AppNavigationState with a narrow history destination that can represent an exercise history target by linked exercise identifier when available, with snapshot-name fallback when needed. The sheet’s full-history action dismisses the sheet, sets selectedTab to .history, sets historyMode to .exercises, and asks HistoryView to open ExerciseHistoryDetailView for the matching exercise.

Quick Sheet Content

  • Title: current exercise snapshot name.
  • Recent sessions: up to three completed workout sessions for the exercise, newest first.
  • Each session group shows workout title, compact date, completed set count, and completed set summaries.
  • Exercise notes: show the historical LoggedExercise.notes under that session’s completed sets when the note is non-empty after trimming whitespace.
  • Empty state: show a read-only empty state when no completed history exists for the exercise.
  • Full-history action: show only when a matching history summary exists.

Reuse Strategy

The sheet should have its own container and layout because it is optimized for quick reference during a workout. It should not embed ExerciseHistoryDetailView directly.

Reuse or lightly extract shared rendering where that prevents formatting drift. Good candidates include set summary formatting, completed set count labels, session group card content, compact date/title display, and exercise-note presentation. If the existing detail view is too page-specific, introduce a small shared component such as ExerciseHistorySessionGroupCard and use it from both the full history detail page and the quick sheet.

Data Rules

  • Only completed WorkoutSession records contribute to exercise history.
  • Only completed LoggedSet records appear in history groups.
  • The active workout must not appear in the quick sheet.
  • Matching should prefer Exercise.id when present.
  • When the exercise link is missing, matching should fall back to case-insensitive exerciseSnapshotName.
  • Historical display names should come from the logged exercise snapshot, preserving the name used at logging time.

Error And Empty States

  • If there are no completed matching sessions, show an empty state such as “No History Yet” with copy explaining that completed workouts for this exercise will appear here.
  • If a deep-link target no longer has matching history by the time the History tab opens, the app should show the normal History empty or list state rather than crashing.
  • The quick sheet is read-only, so most persistence failures are out of scope. Query failures should degrade to an empty or error state consistent with existing app components if SwiftData exposes one in implementation.

Testing Plan

  • Unit-test quick-history grouping so it includes only completed sessions and completed sets.
  • Unit-test exercise matching by linked exercise identifier.
  • Unit-test fallback matching by case-insensitive snapshot name when the exercise link is missing.
  • Unit-test the three-session cap and newest-first ordering.
  • Unit-test exercise notes visibility: non-empty notes appear, blank or whitespace-only notes do not.
  • Test navigation state behavior for switching to History, selecting Exercises mode, and targeting the requested exercise detail.
  • Add focused UI coverage if practical for the header button, sheet presentation, empty state, and full-history action.
